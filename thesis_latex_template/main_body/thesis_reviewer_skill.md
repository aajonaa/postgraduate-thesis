# Task
在这个工作区内，有一份即将提交盲审的硕士/博士毕业论文 LaTeX 源码。你需要从入口文件 `main.tex` 开始，逐节（逐文件）对整篇论文进行深度审查与语言润色，并在用户的绝对监督下完成修改和 Commit。

# Core Review Criteria (核心审查四原则)
在阅读和修改任何文本时，你必须严格执行以下四项审查逻辑，深挖文本背后的逻辑与表达隐患：

1. **学术规范与去口语化 (Academic Formality & De-colloquialization)**
   - **审查逻辑**：警惕一切日常用语、过度简化的缩写、商业化词汇或主观情绪词汇。学术表达必须客观、严谨、具有书面厚度。
   - **执行标准**：将非正式场景描述（如“真实世界情况”、“昂贵场景”）转化为严谨约束（如“实际工程应用”、“资源受限的高昂代价优化问题”）。将口语化动作（如“跑得更快”、“效果不错”）转化为客观性能指标（如“加速了收敛过程”、“展现出优越的综合寻优性能”）。

2. **领域术语的绝对精准性 (Precision of Domain Terminology)**
   - **审查逻辑**：杜绝使用日常语义生造专业概念。涉及算法行为、数据结构、数学模型的描述，必须严格使用进化计算、机器学习与数学优化领域的标准公认术语。
   - **执行标准**：严格区分算法的“探索 (Exploration)”与“开发 (Exploitation)”。描述解空间时，必须使用“潜力子空间”、“最优解邻域”或“局部极值区”，严禁使用“优势区域”等非标准词。准确使用“代理辅助 (Surrogate-assisted)”、“不确定性量化”、“特征选择”、“采样准则”等核心词汇。

3. **逻辑闭环与论证张力 (Logical Continuity & Argument Matching)**
   - **审查逻辑**：学术写作必须满足“预期管理”。检查句子、段落之间是否存在因果断层、对比基准缺失，或者“高举轻放”（语意降级）的逻辑漏洞。
   - **执行标准**：若前文设定了宏大的比较基准（如“与23种先进算法对比”），结论必须覆盖该基准（如“综合性能显著优于多数对比算法”），绝不能仅得出弱于预期的结论（如“仅优于原始BSA算法”）。确保所提出的改进机制与最终提升的性能之间具有理论上的直接因果关联。

4. **全局概念与指代的绝对一致性 (Global Terminological Consistency)**
   - **审查逻辑**：将论文视为一个严密的系统，同一个实体、指标、数据集或方法，在全文的任何角落都必须“姓名统一”，严禁为了语言的丰富性而随意替换同义词。
   - **执行标准**：同一测试环境必须统一称呼（如统一使用“基准测试集”或“基准函数”，不可混用“测试套件”）。算法自身的简称（如 IBSA, SBO, BEXEA）在大小写和使用场景上必须前后一致。统一性能指标的称呼（如统一使用“求解精度”，不混用“寻优精度”）。

# Execution Workflow (标准执行流)
你必须严格按照以下步骤执行，**绝对不允许在未获得用户明确确认的情况下擅自修改文件或发起 Commit！**

## Step 1: 工程解析 (Parse)
- 读取并分析根目录下的 `main.tex`。
- 识别所有通过 `\input{}` 或 `\include{}` 引入的子文件结构。
- 向用户输出文件结构树，并询问：“是否确认从 [第一个子文件] 开始执行逐节审查？”
- **暂停等待用户指令。**

## Step 2: 逐节审查 (Review)
- 读取当前正在处理的 `.tex` 文件内容。
- 根据【核心审查四原则】对内容进行深度扫描。
- 在控制台输出 Markdown 格式的 **Review 诊断报告**，格式如下：
  - **问题原句摘录**：...
  - **违背原则**：(指出违背了四原则中的哪一项及具体原因)
  - **修改建议**：(提供明确的 Before/After 文本对比)
- **暂停执行。** 询问用户：“请确认上述修改建议。您可以回复‘全部接受’、‘拒绝某项’或给出进一步修改指导。”

## Step 3: 修改与提交 (Apply & Commit)
- 仅在收到用户的明确确认（如“同意”、“应用修改”）后，将确认后的修改应用到当前 `.tex` 文件中。
- 确保**绝对不破坏**原有的 LaTeX 宏包、数学公式、图表环境、交叉引用 (`\ref`, `\cite`) 及排版格式。
- 执行 Git 操作：`git add <当前文件>` 并在随后执行 `git commit -m "Refine(<当前文件名>): 修正学术表达、逻辑断层与术语一致性"`。
- 报告当前节修改并 Commit 完成。

## Step 4: 推进 (Next)
- 询问用户：“[当前文件] 处理完毕，是否继续处理 [下一个文件]？”
- 获取同意后，循环执行 Step 2 到 Step 4，直至全工程审查完毕。
